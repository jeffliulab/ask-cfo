# Ask CFO 产品需求文档（PRD v0.1 草稿）

> 状态：v0.1 起步草稿。从 fin-pilot v0.1.0 复制框架壳子之后，CFO 业务部分需要
> 全部从零设计。本文是初版骨架，**多处留待用户决策**（每节末有"待你定"清单）。
> 用途：作为开发依据 + 体现产品判断。

---

## 1. 一句话定位

Ask CFO 是面向**中国小微企业 / 个体工商户 / 代账机构**的 AI 财务工作台，
对标 Cursor 之于程序员、fin-pilot 之于金融分析师 —— 让财务人员的"凭证录入 / 月结 /
报税 / 法规检索"日常被 AI 加速 80%，并保留**完整法规出处可追溯**。

**类比**：
- Cursor 之于程序员
- fin-pilot 之于金融分析师
- Ask CFO 之于财务

## 2. 形态（沿用 fin-pilot 已验证的"金融版 Cursor"形态）

```
┌──────────────┬─────────────────────────────────┬──────────────┐
│ 左：模块菜单  │ 中：工作区 (Workspace)            │ 右：聊天      │
│              │                                  │              │
│ ▸ 凭证录入   │ [拖入发票 / 银行流水]              │ "这笔餐饮能  │
│ ▸ 月结对账   │ ┌──────────────────────────────┐ │ 抵进项税吗？"│
│ ▸ 财务报表   │ │ 凭证草稿卡（借方/贷方）        │ │              │
│ ▸ 报税申报   │ │ 关联准则卡（CAS 第 X 号）      │ │ Agent: 不能  │
│ ▸ 法规问答   │ │ 风险提示卡（业务招待费税前限额）│ │ + [1] [2]    │
│              │ │                              │ │              │
│ ──────────── │ │ 引用 [1] → 抽屉打开 PDF 原文  │ │             │
│ 历史会话     │ └──────────────────────────────┘ │             │
└──────────────┴─────────────────────────────────┴──────────────┘
```

### 三个非协商性设计原则（沿用自 fin-pilot）

1. **Workspace 是一等公民，对话只是触发器** —— 每次 AI 回答必须沉淀成 workspace
   卡片（凭证草稿 / 对账表 / 税额计算单 / 法规摘要），不能只在对话流里飘走
2. **公司主页模板化 + Generative Grid 横向比较** —— 凭证类型固定模板（费用 /
   折旧 / 收入 / 进项 ...），代账场景支持"多客户 × 多期"网格批量出表（v0.6+）
3. **引用即抽屉（Citations as Drawer）** —— inline 角标点击 → 侧抽屉打开**法规
   原文 / 准则条款 / 国税总局公告 PDF**，不离开 workspace
   → **CFO 场景这条价值翻倍**：财务每天都要查"这条政策怎么说"，金融人偶尔查

## 3. v0.1 模块范围

| # | 模块 | v0.1 候选优先级 | 描述 |
|---|------|------|------|
| 1 | **凭证录入** | ⭐ **P0 候选** | 上传发票/银行流水 → AI 草拟分录 + 引用准则 |
| 2 | 月结对账 | ⏳ v0.2 | 凭证审核 + 试算平衡 + 结转 |
| 3 | 财务报表 | ⏳ v0.3 | 三大报表 + 多期对比 + 比率 |
| 4 | 报税申报 | ⏳ v0.4 | 增值税 / 所得税 / 个税自动算 + 预填表 |
| 5 | **法规问答** | ⭐ **P0 候选** | RAG 检索税法 + 准则 + 公告，引用原文 |

**v0.1 推荐组合**：**凭证录入 + 法规问答**。理由：
- 两者都是**单次交互可演示**（不需要等月底才有数据）
- 两者**共享 Citation drawer**（凭证里的"准则依据"和法规问答的"条款原文"是同一形态）
- 法规问答的 RAG 数据可被凭证录入复用
- **演示故事最强**：上传发票 → AI 出凭证 + 引用准则 → 用户问"为啥不能抵扣？" → 跳转抽屉看原文

> ⚠️ **待你定**：v0.1 是 1 个模块还是 2 个？1 个的话哪个先（凭证 vs 法规）？

## 4. 各模块详细设计

### 4.1 凭证录入（v0.1 候选）

**用户场景**：小老板拍发票照片传给会计，会计每月录入几百张凭证。

**输入**：
- 发票图片（PDF / JPG / 微信截图）
- 业务说明文字（"今天请客户吃饭，5 个人 530 元"）
- 银行流水截图（v0.2 起）

**AI 产出**（中区 workspace）：
- **凭证草稿卡**：借方 / 贷方 / 摘要 / 金额
- **税务标记卡**：是否可抵扣进项 / 税前扣除限额 / 适用税率
- **准则依据卡**：相关 CAS 条款 / 增值税细则 / 国税总局公告

**待你定（关键 v0.1 决策）**：
- [ ] **OCR provider**：Azure Document Intelligence / 腾讯云 OCR / 自部署 PaddleOCR / 用大模型直接看图（GPT-4o vision）？
- [ ] **会计准则数据源**：是否商业数据库订阅，还是爬取公开的财政部/税总公告？爬取的合规风险？
- [ ] **凭证草稿编辑流**：用户能否直接在卡片上改 → 入库？v0.1 是只读演示还是支持编辑？
- [ ] **多企业**：v0.1 假设单一公司账套，还是支持代账场景的多客户切换？

### 4.2 月结对账（v0.2）

**用户场景**：每月初做上月结账，需要审核全部凭证 + 试算平衡 + 结转损益。

**待 v0.2 决策**：审核维度（科目 / 金额异常 / 重复入账）；结转规则的可配置性。

### 4.3 财务报表（v0.3）

**用户场景**：导出资产负债表 / 利润表 / 现金流量表给老板 / 银行 / 税局。

**待 v0.3 决策**：报表口径（一般纳税人 vs 小规模 vs 个体户）；多期对比格式；图表渲染（Recharts 沿用 fin-pilot 经验）。

### 4.4 报税申报（v0.4，⚠️ 高风险高价值）

**用户场景**：每月 / 每季度 / 每年向国税局电子税务局申报。

**重要风险**：
- ⚠️ **法律责任**：申报错误老板担责；agent 给的建议算"咨询"还是"代办"？
- ⚠️ **执照**：代理记账业务在中国需要"代理记账许可证"
- ⚠️ **国税系统改版**：电子税务局界面 / 接口频繁更新

**legacy/wencfo/tax_service 有 Playwright 浏览器自动化方案**作参考（Selenium 网页录制操作），但生产可用性需重新评估。

**v0.4 启动前必须做**：
- [ ] 法律咨询：界定服务边界（"辅助决策" vs "代客申报"）
- [ ] 牌照评估：是否需要代理记账资质
- [ ] MVP 范围：先做"申报表预填" + "数字校验"（用户最后手动点提交），不直接代客申报

### 4.5 法规问答（v0.1 已锁 ⭐）

**用户场景**：会计在工作中遇到不确定的处理（"研发费用怎么加计扣除？"），希望快速查到准确条款。

**输入**：自然语言问题
**AI 产出**：
- **答案卡**：直接结论（含 [N] 角标）
- **条款摘要卡**：相关条款的精简引用
- **agent trace（v0.1 新增）**：折叠面板显示 agent 的 search / get 调用轨迹，让用户看到推理过程
- **citation drawer**：点 [N] 抽屉打开**法规原文**（v0.1 外链官网；v0.5 PDF.js 高亮）

**v0.1 实施方案（2026-04-25 锁定）**：
- **数据源**：手工整理 30–50 条最常用条款 → `backend/data/regulations/seed.yaml`（增值税 + 企业所得税 + 个税 + 6–8 个核心 CAS 准则）。爬虫 / 商业数据库推到 v0.5+。
- **检索**：**agent search 而非 RAG**。BM25 keyword 命中（`rank_bm25` over `title + summary + tags`，纯 python 无 DB）+ LLM 通过 tool_use 多轮自主检索。**v0.1 不引入 chromadb / qdrant / embedding model**。
- **chunk 粒度**：一条 = 一个 chunk（不切分），引用粒度精确到条款编号。
- **覆盖范围**：v0.1 先做增值税 + 企业所得税 + 个税三大主税 + CAS 核心；其余 v0.5 扩种。

具体任务清单见 [`NEXT_STEPS.md`](NEXT_STEPS.md) Day 1–7。

## 5. v0.1 范围之外（明确不做）

- ❌ 月结 / 报表 / 报税模块（v0.2-0.4）
- ❌ 工作区持久化（v0.4 上 SQLite 一并做）
- ❌ 多用户 / 多企业 / 代账机构权限隔离（v0.6）
- ❌ PDF.js 原文高亮（v0.5）
- ❌ Docker / 部署 / CI（v0.7）
- ❌ 直接对接金税系统 / 电子税务局申报（v0.4 评估）
- ❌ 替代会计师 / 出具审计意见（永远不做，合规红线）

## 6. 数据源 + 合规

### 6.1 法规 / 准则数据源（v0.1 必须先选定）

| 来源 | 合规 | 可行性 | 评价 |
|---|---|---|---|
| 全国法规数据库（chinalaw.gov.cn）| ✅ 官方 | ❌ 无开放 API；爬取要小心反爬 | 权威但工程负担重 |
| 国税总局（chinatax.gov.cn）| ✅ 官方 | ⚠️ 公告分散在不同栏目 | 增值税 / 所得税公告必查 |
| 北大法宝 / 威科先行 | ✅ 商业 | ❌ 个人开发者贵 | 不适合 v0.1 |
| 财政部官网 | ✅ 官方 | ⚠️ 散 | 会计准则原文必查 |
| 第三方爬虫站 | ⚠️ 灰 | ✅ 易 | v0.1 演示可接受，正式不行 |

**v0.1 建议起手**：手动整理 30-50 条**最常用税务/会计法规**做种子数据（HTML/PDF 链接 + 关键条款摘要），先把 RAG 流程跑通；爬虫数据等 v0.5 真做用户时再补全。

### 6.2 OCR provider（v0.1 必须先选定）

| 选项 | 准确率 | 成本 | 易用性 |
|---|---|---|---|
| Azure Document Intelligence | 高 | $$ | API 易用，发票模型现成 |
| 腾讯云 OCR | 高 | $ | 国内调用稳；增值税专票模型现成 |
| 阿里云 OCR | 高 | $ | 同腾讯 |
| GPT-4o vision | 高 | $$$ | 灵活但贵；中文标准发票格式不一定最优 |
| Claude Sonnet vision | 高 | $$ | 同上 |
| 自部署 PaddleOCR | 中 | 免费（自己机器） | 部署麻烦但 0 成本 |

**v0.1 建议起手**：腾讯云 / 阿里云增值税发票识别 API（中文场景成熟、几分钱一张），后期可加 GPT-4o vision 做兜底（识别非标准发票）。

## 7. 与 fin-pilot 的关系

**复用**：
- 三栏壳（ThreePaneLayout / LeftMenu / WorkspaceCanvas / ChatPanel / CitationDrawer）
- LLM 4-provider 抽象 + chat orchestrator + Vercel AI SDK Data Stream Protocol
- useChatStream hook + Citation drawer 形态
- 三层 backend 架构 + 版本追踪三件套 + 端口规范

**不复用**：
- 业务模块（金融 ≠ 财务）
- 数据 provider（AKShare/EDGAR ≠ OCR/法规 RAG）
- 卡片类型（财务 KPI ≠ 凭证草稿）
- prompt 模板（投研分析师 ≠ 会计）

**未来共享代码**：等两个项目都进 v0.4+，UI primitives 真的稳定后，考虑抽
`agent-shell-ui` npm 包 + `agent-llm-providers` pip 包供两边引用。**v0.1 阶段
代码各自维护，演化清晰再合并**。
